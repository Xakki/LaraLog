# Правила логирования

> Прагматичные правила для продакшен-логирования. Простым языком, без воды.
> LaraLog - референсная реализация для PHP/Laravel; ссылки на код идут как `LaraLog: src/...`.
>
> Статус: **draft v1**, итерируем в открытую.
>
> *English version: [LoggingRules.md](./LoggingRules.md)*

## 0. Для кого и зачем

Для инженеров и тимлидов, которым нужно, чтобы логи:

- **Позволяли определять причину проблемы** - разработчик или дежурный мог оперативно по логам понять контекст и причину ошибки;
- **Были дешёвыми и надёжными** - инфраструктура логирования, затраты на сбор, баланс между гарантией доставки, актуальностью и ресурсозатратами, пайплайны, индексы и retention - это реальные деньги;
- **Обеспечивали безопасность** - никаких PII, секретов и сессий в долгое хранение;
- **Сцеплялись с трейсами и метриками** - одна транзакция, одна история, возможность выстроить цепочку событий.

Описанные принципы независимы от языка программирования. PHP-конкретика вынесена в блоки на примере этой библиотеки.

## 1. Базовые принципы

1. **Лог - это API.** Им пользуется oncall, ops и ты сам через полгода. Сломать имя поля так же больно, как сломать REST. Каталог полей и процесс переименования - см. §9.3.
2. **Структура важнее прозы.** Запись лога - JSON-объект со стабильными именами полей. `message` - для глаз. Всё, по чему ищут, - в `context` и `extra`.
3. **Cardinality - не забывать об этом.** То, что меняется каждый запрос (`request_id`, `user_id`, `order_id`), - золото в полях, яд в индексных лейблах. См. §5.5.
4. **Логируем один раз, на границе.** Сбой пишется один раз - там, где есть полный контекст. Catch-log-rethrow на каждом уровне = умножение шума без новой информации.
5. **Стабильность - спринтовый налог.** Логированию, как части техдолга, нужно уделять время: ratchet в CI, ревью по расписанию, удаление того, что никто не читает, настройка алертов и дашбордов.
6. **Логирование не должно ронять запрос.** Если Graylog лежит или Redis полный - запрос всё равно завершается успешно. Логирование best-effort.

## 2. Что мы НЕ логируем

Не обсуждается. Нарушение - это security-инцидент, а не вопрос стиля.

| Никогда                               | Примеры                                        | Почему                        |
|---------------------------------------|------------------------------------------------|-------------------------------|
| Пароли, хеши, токены восстановления   | `password`, `password_hash`, `reset_token`     | Тривиальная кража credentials |
| API-ключи, bearer/refresh-токены      | `Authorization: Bearer …`, `X-Api-Key`         | То же                         |
| Сессионные ID                         | `PHPSESSID`, `JSESSIONID`, `Cookie: session=…` | Перехват сессий               |
| Полные платёжные данные               | PAN, CVV, срок                                 | PCI-DSS                       |
| Государственные ID                    | СНИЛС, ИНН, паспорт, SSN                       | GDPR/HIPAA                    |
| Медицинские данные                    | Диагноз, лекарства, биометрика                 | HIPAA                         |
| Полный PII без необходимости          | Email + телефон + адрес вместе                 | GDPR data minimization        |
| Сырые body эндпоинтов с любым из выше | `POST /login`, `POST /payment`                 | Всё перечисленное             |

**Редактируем на границе процесса, не в хранилище.** Если значение чувствительное - его не должно быть в потоке. Безопасные плейсхолдеры: `***`, `[redacted]`, `sha256:<первые-8-символов>` если нужно трассировать без раскрытия.

Внимательно с дампами HTTP, трейсами с аргументами, биндингами ORM, сообщениями исключений, в которые подставляется user input. LaraLog уже урезает строки `bindings` >512 chars и показывает их только при `APP_DEBUG=true` - в проде это включать нельзя.

### 2.1 Безопасность самого пайплайна

Редакция значений на источнике - это только половина. Сам пайплайн логов - такой же attack surface, как и любой сервис.

**Log injection / log forging.** Если в поле попадает user input с `\n`, `\r` или ANSI-escape, атакующий может подделать строки лога: вставить фальшивую запись, чтобы скрыть следы или обвинить чужого пользователя. Защита:

- На источнике: JSON-сериализация (она экранирует control chars). Никогда не клеить лог конкатенацией строк.
- Для plain-text транспортов (syslog line format): экранировать `\n`/`\r` явно перед записью.
- На приёмнике (Graylog/Loki): валидировать схему, дропать строки без обязательных top-level полей.

**Транспорт.** UDP plain text допустим **только внутри trusted network** (private VPC, k8s pod-to-pod). На границе сегмента или через интернет - syslog over TLS (RFC 5425), RELP с TLS, или HTTPS-вход в коллектор. Audit-канал - всегда TLS, независимо от сегмента.

**Encryption at rest.** Operational - на усмотрение провайдера хранилища (обычно server-side encryption на S3/диске - достаточно). Audit и любое хранилище с PII - явное требование, customer-managed keys где регулятор обязывает.

**Доступ на чтение операционных логов - это сам по себе аудит-событие.** Не только audit-канал (§6.5.7). Запросы инженеров к prod-логам через UI (Graylog, Kibana, Loki Explore) пишутся как audit:

```jsonc
{"action":"logs.operational.read","actor":{"user_id":..., "role":"engineer"},
 "target":{"stream":"prod-api","query":"...","time_range":"..."},
 "result":"success"}
```

Особенно важно для окон, где могли быть PII (даже если редакция должна была сработать). Защита от инсайдера и регулятор-friendly.

**Изоляция между средами.** Логи dev/staging не сваливаем в тот же индекс/стрим, что prod - кроссзагрязнение поиска и алертов. Один пайплайн, разные потоки.

## 3. Уровни

8 уровней PSR-3 / RFC 5424. На практике активно используем 6: `debug`, `info`, `notice`, `warning`, `error`, `critical`. `emergency` - частный случай critical (всё легло). `alert` обычно сворачивается в `critical`.

### 3.1 Дерево решения

```
Сервис обслуживает запросы?
├─ Нет, лежит всё                                              → EMERGENCY
└─ Да
   ├─ Данные повреждены или потеряны?                          → CRITICAL  (немедленный алерт)
   ├─ Операция упала, восстановления нет, ушло к пользователю? → ERROR     (алерт от 1% трафика)
   ├─ Сбой поглощён, но НУЖНО что-то сделать?                  → WARNING   (алерт от 5%)
   ├─ Странное / подозрительное / автокорректировалось?        → NOTICE    (алерт от 10%)
   ├─ Нормальное бизнес-событие?                               → INFO
   └─ Полезно только при воспроизведении бага?                 → DEBUG     (в проде выключен по умолчанию, включать только с условием)
```

### 3.2 Что такое NOTICE

Notice - это "обратить внимание, если станет много". Не алертит на отдельное событие, алертит на тренд.

Сюда попадают:

- **Ошибки типов и валидации**, которые система обработала: пришёл `string` вместо `int` и был приведён, пустое поле заменено дефолтом, неизвестный enum-кейс схвачен `default`.
- **Подозрительные действия пользователей**: 3 неуспешных логина за минуту от одного IP, попытка изменить чужой ресурс (получил 403, но факт стоит зафиксировать), запрос с подозрительными заголовками.
- **Сбои, которые "должны бы быть" Warning, но**:
  - данные автокорректируются и работа продолжается;
  - на ход выполнения это не влияет;
  - для них нет конкретного действия со стороны разработчика.
- **Deprecated-пути**: вызвали старый API, но он отработал.
- **Первая попытка retry** после сбоя upstream (см. §6.4).

Скачок Notice выше нормы - сигнал, что что-то изменилось. Стабильный фон Notice - норма.

### 3.3 WARN ↔ NOTICE: правило понижения

**Если для Warning нет конкретного действия - это Notice.**

Тест: "Если разработчик увидит этот warning, что он должен сделать?" Если ответ "ну, иногда так бывает" - переводим на notice. Алертить на warning без actionability - гарантированный warning fatigue.

### 3.4 WARN vs ERROR (самое нарушаемое)

- **WARN = система поглотила сбой, но что-то пошло не так.** Сработал fallback, заполнили из кеша, дёрнули старый путь. Запрос пользователя завершён успешно.
- **ERROR = сбой ушёл наружу.** Все retry'и исчерпаны, операция не завершилась, пользователь получил 5xx или неверный результат.

Следствие для retry-цикла: промежуточные попытки → `warning`, финальное исчерпание → `error`. См. §6.4.

### 3.5 Алерты по уровням

Числа - стартовые, тонкая настройка под трафик сервиса.

| Уровень              | Порог               | Условие                                                          |
|----------------------|---------------------|------------------------------------------------------------------|
| Critical / Emergency | **Сразу**           | любое одно событие                                               |
| Error                | **> 1% всего лога** | `rate(level=error)[5m] / rate(*)[5m] > 0.01` AND `count >= 5`    |
| Warning              | **> 5%**            | `rate(level=warning)[5m] / rate(*)[5m] > 0.05` AND `count >= 20` |
| Notice               | **> 10%**           | `rate(level=notice)[5m] / rate(*)[5m] > 0.10` AND `count >= 50`  |
| Info / Debug         | без алертов         | -                                                                |

**Знаменатель `rate(*)`** = общий поток логов сервиса. Универсален, но смещён в сторону info-тяжёлых сервисов: если ты пишешь много `info` (например, по §6.4.1 - на каждый успешный внешний вызов), знаменатель растёт, и `error` rate в процентах падает. Не критично - floor count страхует, - но при сравнении сервисов цифры нужно интерпретировать с поправкой на verbosity.

**Floor count** (`>= N`) защищает от ложняков на низком трафике: 1 событие из 5 = 20%, но oncall будить не за что.

**Окно 5 минут** - компромисс: достаточно длинное, чтобы среднее не дёргалось от выбросов, достаточно короткое, чтобы заметить активный инцидент. Burst-сценарий (50 ошибок за 30с + тишина) дополнительно ловится правилом `increase(level=error)[1m] >= 20` параллельно с rate-порогом.

**Оси разреза - ставим минимум две:**

| Ось         | Что ловит                                            | Пример                                                                |
|-------------|------------------------------------------------------|-----------------------------------------------------------------------|
| global      | системные сбои                                       | `rate(level=error)[5m] / rate(*)[5m]`                                 |
| per `tag`   | поломку одного домена/функции                        | `rate(level=error{tag="billing"})[5m] / rate(*{tag="billing"})[5m]`   |
| per route   | поломку одного эндпоинта                             | то же с `request_url` (только не как Loki label - см. §5.5)           |
| per `tier`  | различение фронт/worker/cron                         | `rate(level=error{tier="worker"})[5m] / rate(*{tier="worker"})[5m]`   |

Global без per-tag даёт частые false negatives: 0.5% ошибок глобально могут быть 30% по конкретному tag'у `billing`, что уже инцидент.

### 3.6 Дефолт в продакшене

- Уровень логирования прода: `info`.
- Hot path с >10 INFO/req - это `debug`.
- DEBUG включается локально, per-instance, по специальному запросу, для конкретного инцидента; никогда глобально.

### 3.7 Трейс и точка вызова

Трейсы нужны, чтобы понять, где вызвался лог или исключение. Но это объёмные данные.
Для любых логов важно знать, где он создаётся (для быстрой отладки).
На уровне `Warning` и выше автоматически добавляется стек-трейс. Глубина трейса зависит от уровня - чем серьёзнее, тем больше трейса. То же касается и контекста лога (в критических логах контекст может иметь решающую роль):

| Уровень                      | Глубина трейса (кадров) |
|------------------------------|-------------------------|
| Warning                      | 5                       |
| Error                        | 10                      |
| Critical / Alert / Emergency | 20                      |

`src/LogManager.php` принимает все 8 PSR-3 методов.
Vendor/Monolog/Illuminate кадры вырезаются (`TraitFileTrace`). Глубина, список вырезаемых путей и лимит на аргумент настраиваются в `config/logger.php` (`trace.depth`, `trace.excluded_partials`, `trace.arg_limit`).

## 4. Форма лога

### 4.1 Top-level поля (контракт)

Фиксированный набор, ничего своего сюда не добавляем - это публичный контракт лога / Monolog совместимый.

| Поле         | Тип        | Заметка                                                                 |
|--------------|------------|-------------------------------------------------------------------------|
| `datetime`   | RFC 3339   | `Y-m-d\TH:i:s.uP` `2026-05-28T14:23:45.123456+00:00`                    |
| `level`      | int        | severity_number RFC 5424 / PSR-3                                        |
| `level_name` | string     | имя уровня в lowercase                                                  |
| `channel`    | string     | имя канала PSR-3/Monolog (`app`, `sql`, `audit`, ...)                   |
| `message`    | string     | короткое предложение, без интерполяции значений                         |
| `context`    | array      | значение **зависит от события** (request, user, exception, ресурсы)     |
| `extra`      | array      | значение **стабильно за процесс** (env, релиз, host, версия библиотеки) |

`level` - 100=debug, 200=info, 250=notice, 300=warning, 400=error, 500=critical, 550=alert, 600=emergency. Удобен для числовых фильтров и сортировки (`level >= 300`).
`level_name` - `debug`, `info`, `notice`, `warning`, `error`, `critical`, `alert`, `emergency`. Для глаз и runbook'ов

### 4.2 extra - стабильное за процесс

Заполняется автоматически процессорами при старте процесса. Значение не меняется от вызова к вызову внутри одного запроса/скрипта.

| Поле             | Тип    | Источник                                                       |
|------------------|--------|----------------------------------------------------------------|
| `app_name`       | string | env                                                            |
| `app_env`        | string | env (`prod`, `staging`, `dev`)                                 |
| `app_ver`        | string | release version                                                |
| `log_ver`        | string | версия библиотеки логирования                                  |
| `host_ip`        | string | IP **хоста/ноды**, прокидывается извне (env, k8s downward API) |
| `host_name`      | string | hostname хоста (или container id, если нет внешнего инжекта)   |
| `container_name` | string | docker/k8s                                                     |
| `tier`           | string | `web`/`api`/`worker`/`cron`                                    |
| `release_tag`    | string | git tag или CI build id                                        |
| `release_time`   | string | время деплоя, RFC 3339                                         |
| `console_argv`   | array  | argv для CLI-скриптов                                          |
| `load_avg_1m`    | float  | system load 1 min (`LoadAverageProcessor`)                     |
| `load_avg_5m`    | float  | system load 5 min                                              |
| `load_avg_15m`   | float  | system load 15 min                                             |

> **LaraLog implementation:** `ExtraProcessor` (`src/Processor/ExtraProcessor.php`) + `BaseContext` (`src/BaseContext.php`). `host_ip` берётся из env (например, k8s downward API `status.hostIP`), не через `gethostbyname()` - чтобы значение было реально IP ноды, а не контейнера. Сейчас `request_id` кладётся в `extra` - это legacy; миграция в `context` запланирована (см. §4.3).

### 4.3 context - per-event

Заполняется кодом приложения или авто-injectom через MDC/scope. Меняется на каждый запрос/событие.

| Поле                        | Тип           | Когда                                                                 |
|-----------------------------|---------------|-----------------------------------------------------------------------|
| `log_type`                  | string        | кто инициировал лог; см. §4.3.1 (в `context`, а не top-level, ради Monolog-совместимости) |
| `request_id`                | UUID v4       | из `X-Request-ID` или сгенерирован; переносится через очереди и исходящие вызовы |
| `request_url`               | string        | путь (без query string или с redacted query)                          |
| `request_method`            | string        | `GET`/`POST`/...                                                      |
| `request_host`              | string        | `Host` header                                                         |
| `request_user_agent`        | string        | `User-Agent`                                                          |
| `trace_id`, `span_id`       | hex           | при OTel; `request_id` - вырожденная форма                            |
| `user_id`                   | int           | актор операции                                                        |
| `session_id`                | string (hash) | сессия HTTP / queue worker'а                                          |
| `file`, `line`              | string        | `path:int` авто из стека (`TraitFileTrace`), руками не передаём       |
| `trace`                     | string        | стек-трейс, обрезанный по уровню (см. §3.7)                           |
| `exception`                 | class FQN     | полное имя класса исключения                                          |
| `exception_message`         | string        | Message исключения                                                    |
| `exception_code`            | int           | Code исключения                                                       |
| `exception_prev`, `_prev_*` | object        | previous-цепочка исключения                                           |
| `memory_usage`              | int (байты)   | `memory_get_usage(true)`                                              |
| `memory_peak`               | int (байты)   | `memory_get_peak_usage(true)`                                         |
| `tag`                       | string        | routing-лейбл (`sql`, `queue`, `auth`, ...); whitelist см. §9.3       |
| business IDs                | int / string  | `order_id`, `payment_id`, любое доменное                              |

> **LaraLog implementation:** `LogManager::appendContext()` собирает exception-поля; `LogManager::contextTypeCorrector()` приводит типы.

**Грей-кейс `request_id`:** значение стабильно **внутри одного запроса**, но меняется от запроса к запросу - поэтому `context`, не `extra`. Правило "стабильно за процесс" = "одно значение от старта до выхода процесса/скрипта".

#### 4.3.1 log_type

Помогает в одной куче логов отделить "что писалось из кода намеренно" от "что свалилось само". Без этого поля fatal-сбой теряется среди обычных error'ов в поиске - алертить на "только настоящие падения" становится невозможно.

| Значение    | Источник                                      | Пример (PHP)                                                                             |
|-------------|-----------------------------------------------|------------------------------------------------------------------------------------------|
| `logger`    | явный вызов в коде                            | `Log::info(...)`, `$this->logger->error(...)`                                            |
| `trigger`   | инициировано рантаймом языка                  | `set_error_handler` на `E_NOTICE`/`E_WARNING`, `trigger_error()`, deprecation warnings   |
| `exception` | исключение - uncaught или явно залогированное | `set_exception_handler`, явный `Log::error('...', ['exception' => $e])` на пойманном     |
| `fatal`     | падение скрипта рантаймом                     | Ловиться в `register_shutdown_function`, parse-error, OOM, превышение max_execution_time |

> **LaraLog implementation:** `log_type` проставляется автоматически source-сайтом (exception-handler → `exception`, shutdown → `fatal`, error-handler → `trigger`, остальное → `logger`). Кандидат на v1.x.

Дефолт - `logger`. Типичное условие алерта: `log_type IN (exception, fatal)` отделяет настоящие сбои от "разработчик решил полайфить warning".

### 4.4 Бизнес-идентификаторы - в context, не в message

```jsonc
// плохо
{"level":"info","message":"order 12345 paid by user 67890 for 49.99 EUR"}

// хорошо
{"level":"info","message":"order paid",
 "context":{"order_id":12345,"user_id":67890,"amount_cents":4999,"currency":"EUR","tag":"billing"}}
```

Хорошая форма ищется (`order_id:12345`, `amount_cents>10000`). Плохая - нет.

**Анти-пример с PII** (формально структурированный, но всё равно неправильный):

```jsonc
// плохо - даже в context
{"level":"info","message":"login attempt",
 "context":{"email":"user@example.com","password":"hunter2","ip":"1.2.3.4"}}

// хорошо
{"level":"info","message":"login attempt",
 "context":{"user_id":67890,"ip":"1.2.3.4","auth_method":"password","tag":"auth"}}
```

Структурированность не отменяет редакцию §2. Пароль - всегда в ban-list; email - выкинуть, оставить `user_id`; raw IP - оценить под GDPR (для логина обычно нужен, для каждого запроса - спорно).

### 4.5 Дисциплина типов

Одно и то же поле, приходящее то как `"123"`, то как `123` - это два столбца в Elasticsearch, краш индексера в Loki, путаница в Grafana.

- `*_id`, `*Id`, `*_cnt`, `*_count`, `*Size` → integer
- `is_*`, `has_*`, `flag*` → boolean
- Деньги → integer в минорных единицах (`amount_cents`), никогда float
- Длительности → integer миллисекунды
- Таймстампы в context → одно из: RFC 3339 строка ИЛИ Unix epoch int. Выбираем по проекту, не смешиваем.

> **LaraLog implementation:** `LogManager::contextTypeCorrector()` делает это автоматически. Reserved keys в `src/const.php`.

### 4.6 Лимиты по размеру

| Лимит | Дефолт | Почему |
|---|---|---|
| `message` | ~3 KB | длинный message = дампят payload, надо в context |
| Вся строка | ~1.4 KB (Graylog UDP), ~64 KB (Loki, но <16 KB) | UDP fragmentation, лимиты индексера |
| Стек-трейс | 5 / 10 / 20 кадров по уровню | см. §3.7 |
| Аргументы в трейсе | 128 chars на аргумент | иначе взрывается на data-heavy методах |

**Стратегия обрезки** (`CustomFormatter::trimLog`): сначала `message`, потом string-значения в `context`/`extra`. Числовые/булевы поля не трогаем. Маркер обрезки - `…`, чтобы потребитель знал.

### 4.7 Именование полей

- `snake_case` везде. Смесь camelCase и snake_case в одном пейлоаде ломает дашборды.
- Префиксы для родственных полей: `db_query`, `db_time_ms`, `db_table`.
- Зарезервированные префиксы: `app_*`, `http_*`, `db_*`, `queue_*`. Проектные ключи - со своим префиксом.

## 5. Контекст и корреляция

### 5.1 Три ID

| ID | Область | Время жизни | Источник |
|---|---|---|---|
| `trace_id` | вся распределённая транзакция | до конца операции, через очереди | OTel |
| `span_id` | единица работы внутри trace | время span'а | OTel |
| `request_id` | один HTTP-запрос | один сервис, один запрос | `X-Request-ID` |

Есть OTel - используем `trace_id`/`span_id`. Нет - `request_id` это MVP-замена.

### 5.2 Бизнес-ID

`user_id`, `order_id`, `session_id`. Переживают один trace - единственный способ ответить "что случилось с заказом 12345 за последние 24ч".

Инжектим в logging scope один раз на entrypoint (паттерн MDC). Руками через каждый вызов не таскаем.

### 5.3 Entrypoints

Каждый entrypoint (HTTP, queue worker, scheduled job, RPC) пишет пару:

```jsonc
{"level":"info","message":"entry","context":{"tag":"entrypoint","entrypoint":"POST /orders","payload_size":1234}}
…работа…
{"level":"info","message":"exit","context":{"tag":"entrypoint","entrypoint":"POST /orders","success":true,"duration_ms":83,"status_code":200}}
```

Без exit не видно, завершилось ли. Без entry нет якоря для request_id.

### 5.4 Пересечение границ

- Исходящий HTTP - пробрасываем `X-Request-ID` (или W3C `traceparent` при OTel).
- Исходящий queue job - сериализуем `request_id`/`trace_id` в payload, восстанавливаем в worker'е.
- БД - не пробрасывается; slow-query логи носят собственный `request_id`, инжектится через session variable или SQL-комментарий.

### 5.5 Бюджет cardinality

Самая частая причина того, что пайплайн логов лежит.

> **Правило:** всё, что меняется per-request, идёт в **поле** (queryable, не индексируется). Стабильное на тысячах запросов - может быть **лейблом** (индексируется).

| OK как лейбл (низкая cardinality) | НЕ OK как лейбл                |
|-----------------------------------|--------------------------------|
| `app_env` (dev/staging/prod)      | `request_id`                   |
| `tier` (web/api/worker)           | `user_id`, `order_id`, `email` |
| `tag` (auth/billing/sql)          | полный URL с query string      |
| `level`                           | datetime                       |

Loki взрывает chunk count, Elastic - mapping, Graylog - индекс стрима. По-разному, но одинаково больно.

> [Grafana - Best practices for logging](https://grafana.com/blog/2022/05/16/all-things-logs-best-practices-for-logging-and-grafana-loki/).

## 6. Особые сценарии

### 6.1 Исключения

Логируем сбой **один раз** - на верхнем уровне обработки, где есть бизнес-контекст. Catch-log-rethrow на каждом уровне - нет.

Обязательные поля:

- `exception` - полное имя класса (не сообщение)
- `exception_code` - integer
- `file:line` точки throw
- `trace` - обрезанный по уровню (5/10/20), framework-кадры вырезаны
- `exception_prev`, `exception_prev_*` - полная цепочка через `getPrevious()`

**Сообщение исключения - под подозрением.** В `$e->getMessage()` подставляется user input (`"User john'; DROP TABLE -- not found"`). Это **источник PII и log injection** одновременно. Опции:

- Хранить class FQN + code, message - только при `app_env != prod` или в санитайзенной форме.
- Или применять тот же редактор, что и к request body (§2).

**Fan-out / параллельная работа.** Если на верхнем уровне запускается `Promise.all` / параллельные jobs и часть упала - логируем **каждую упавшую ветку отдельно**, с общим `trace_id` и индивидуальным `branch_id` или index. Один суммарный лог скрывает причины; отдельные восстанавливаются по `trace_id`.

**Enrichment без логирования.** Если на нижнем уровне есть контекст, которого нет на верхнем (например, `db_query` в Repository, не виден в Controller), - **не логируем там**, а добавляем в MDC/scope (`Log::withContext(['db_query' => ...])`). На верхнем уровне лог соберёт всё.

> **LaraLog implementation:** `LogManager::appendContext()`. `TraitFileTrace` вырезает `Monolog`, `Illuminate/Log/`, `vendor/laravel`.

### 6.2 Медленные SQL-запросы

Отдельный канал, `tag: sql`. Обязательные поля: `db_query` (обрезанный), `db_table`, `db_time_ms`, `sql_type`. **Никогда** не логируем сырые `bindings` в проде.

Два порога:

```
SQL_SLOW_LOG_FOR_SELECT=200ms
SQL_SLOW_LOG_ALL=500ms
```

Write-операции ожидаемо медленнее, у них порог выше.

**N+1 detection.** Отдельный класс slow-query: суммарно много быстрых запросов в одном request_id. Считаем `count(distinct db_query_normalized)` в рамках одного request_id; порог - >50 одинаковых запросов = N+1, `tag: sql_n_plus_1`, `level: warning`. Чем именно нормализовать (substitute literals, AST-fingerprint) - tradeoff между точностью и нагрузкой.

> **LaraLog implementation:** `src/SqlLogServiceProvider.php`. Bindings - только при `APP_DEBUG=true` и ≤ 20 биндов, строки >512 chars заменяются `string(N)`. Telescope-запросы исключены. N+1 detection - кандидат на v1.x.

### 6.3 Очереди и фоновые worker'ы

**Heartbeat - через Prometheus, где это возможно.** Worker экспортирует метрику:

```
queue_worker_heartbeat_total{worker="default"} (counter, инкремент каждые N сек)
queue_worker_last_active_timestamp{worker="default"} (gauge, время последней активности)
```

Алерт срабатывает, когда метрика не обновляется X секунд. Это дешевле, точнее и не засоряет лог-пайплайн.

Heartbeat в лог оставляем там, где Prometheus недоступен (например, разовый scheduled job, у которого нет долгоживущего процесса для scrape'а).

Остальное:
- Job start/finish - пара entrypoint, как в §5.3, `entrypoint: queue:<job-class>`.
- На `WorkerStopping` пишем `info` со status'ом - чтобы видеть, чисто вышел или был убит.

> **LaraLog implementation:** `src/QueueHeartBeat.php` пишет heartbeat в лог (legacy подход). Миграция на Prometheus-метрику - кандидат на v2.

### 6.4 Внешние вызовы и retries

#### 6.4.1 Успешный вызов

**На каждый успешный внешний вызов пишем `info` лог.** Без payload, только метрики и идентификаторы.

```jsonc
{"level":"info","level_name":"info","message":"http call ok",
 "context":{"tag":"upstream","target":"payments-api","method":"POST","path":"/charge",
            "response_code":200,"latency_ms":143,"response_size":512,"request_id":"..."}}
```

Зачем не оптимизируем "молчание - норма":

- p50/p95/p99 latency считаются прямо из логов (или конвертируются в Prometheus histogram - см. §8.2).
- baseline error rate (отношение error/total по `target`) нужен знаменатель.
- "молчание" неотличимо от "запрос не дошёл вообще" - история теряется.

Обязательные поля:

| Поле             | Тип    | Заметка                                   |
|------------------|--------|-------------------------------------------|
| `target`         | string | имя upstream (низкая cardinality, не URL) |
| `method`, `path` | string | для HTTP; path без query string           |
| `response_code`  | int    | HTTP/gRPC код                             |
| `latency_ms`     | int    | end-to-end                                |
| `response_size`  | int    | в байтах                                  |
| `attempt`        | int    | 1 при первой удачной попытке              |

**Что не кладём:** тело request/response, headers с credentials, query string с возможным PII. Если нужна форма тела - `content_type`, `response_size`.

#### 6.4.2 Прогрессия уровней по сбоям

"Первая попытка" в таблице ниже = **первый сбой при вызове** (т.е. первая попытка ретрая, поскольку retry запускается только после провала исходной попытки).

| Событие                                 | Уровень   |
|-----------------------------------------|-----------|
| Первый сбой при вызове (retry стартует) | `notice`  |
| Все последующие попытки retry           | `warning` |
| `max_attempts` исчерпан                 | `error`   |
| Circuit breaker открылся                | `warning` |
| Circuit breaker остался открыт > N мин  | `error`   |

Логика: первый сбой upstream - норма распределённых систем, ожидаемо. Стабильный поток retries - сигнал, что upstream деградирует (порог Warning 5%). Все попытки выгорели - клиент получил отказ (Error).

Обязательные поля на сбое (в дополнение к §6.4.1):

- `attempt` - номер попытки, начиная с 1
- `max_attempts`
- `backoff_ms` - сколько ждали до этой попытки
- `error_reason` - короткая строка (`timeout`, `connection_refused`, `5xx`, `4xx`), не текст
- `exception` (если был) - класс, не сообщение

Idempotency-ключи кладём в context. Тело response для retries не логируем - только форму (`response_size`, `content_type`).

### 6.5 Audit-логи

#### 6.5.1 Что это и зачем отдельно

Audit-лог фиксирует "кто, что, с чем, когда и с каким результатом сделал". Это другой контракт по сравнению с операционным логом:

| | Операционный лог | Audit-лог |
|---|---|---|
| Цель | Дебаг, мониторинг | Compliance, расследования, юр. свидетельство |
| Storage | Хорошо бы, но не критично | Обязательно durable + append-only |
| Retention | Дни-недели | Годы (1-7 в зависимости от регулятора) |
| Доступ к чтению | Любой инженер | Отдельная роль, доступ сам аудитится |
| Сэмплинг | Можно | Никогда |
| Транспорт | Best-effort (UDP допустим) | Synchronous, at-least-once |

**Не смешиваем потоки.** Отдельный sink, отдельный индекс/таблица/bucket, отдельный retention.

#### 6.5.2 Когда нужен

Обязательно, если есть хоть одно из:
- Платёжные данные → PCI-DSS
- Персональные данные ЕС → GDPR Art.30 (Records of Processing)
- Медицинские данные → HIPAA
- Финансовая отчётность → SOX
- Сертификация ISO 27001 / SOC 2
- B2B с DPA (Data Processing Agreements) - практически всегда

#### 6.5.3 Какие события всегда аудитим

| Категория | Примеры |
|---|---|
| Аутентификация | login/logout, неуспешный вход, смена пароля, MFA добавлен/удалён, password reset |
| Авторизация | выдача/отзыв ролей, изменение permissions, impersonation |
| Доступ к данным | чтение/экспорт PII или финансовых данных |
| Изменение данных | CRUD на критичные сущности (пользователи, заказы, биллинг, конфиги) |
| Безопасность | ротация ключей, изменение security-настроек, blocked IP |
| Администрирование | создание/удаление аккаунтов, изменение системного конфига |
| API-ключи / tokens | создание, отзыв, ротация |

#### 6.5.4 Обязательные поля

5W: who, what, where, when, what changed.

```jsonc
{
  "datetime": "2026-05-28T14:23:45.123456+00:00",
  "actor": {
    "user_id": 67890,
    "role": "admin",
    "session_id": "sha256:a1b2c3d4",
    "ip": "1.2.3.4",
    "user_agent": "...",
    "impersonated_by": null
  },
  "action": "user.role.grant",
  "target": {
    "type": "user",
    "id": 12345,
    "owner_id": 12345
  },
  "result": "success",
  "before": {"role": "viewer"},
  "after": {"role": "admin"},
  "request_id": "uuid-...",
  "reason": "ticket #1234",
  "audit_ver": "1"
}
```

- **Who** (`actor`) - id, роль, IP, сессия, был ли это impersonation
- **What** (`action`) - глагол в формате `domain.entity.verb`: `user.role.grant`, `order.delete`, `payment.refund`
- **Where** (`target`) - тип и id ресурса, плюс `owner_id` если actor ≠ owner
- **When** (`datetime`) - серверное время с микросекундами
- **What changed** (`before`/`after` для мутаций) + `result` (success/failure)
- `request_id` - корреляция с operational-логами
- `reason` - заявка, комментарий, обоснование (если бизнес-процесс требует)
- `audit_ver` - версия схемы audit-лога, чтобы выдержать миграции схемы

#### 6.5.5 Immutability

- **Append-only**: писатель имеет только INSERT. Никаких UPDATE/DELETE на уровне приложения.
- На уровне БД - TRIGGER, запрещающий UPDATE/DELETE на этой таблице, или роль с правами только на INSERT.
- WORM-хранилище если требует регулятор: S3 Object Lock, GCS Bucket Lock, Azure Immutable Storage.
- Hash chain для high-compliance (опционально): каждая запись хранит `prev_hash` = sha256 предыдущей. Подделка середины истории становится математически дорогой.

#### 6.5.6 Retention (типовые сроки)

| Регулятор | Минимум |
|---|---|
| PCI-DSS | 1 год, из них 3 месяца hot-access |
| GDPR | по дольшему из бизнес-необходимости и другой регуляции |
| HIPAA | 6 лет |
| SOX | 7 лет |
| ISO 27001 | по политике организации, обычно 3-7 лет |

Сверяемся с юристами под конкретный продукт. Цифры - стартовые ориентиры.

#### 6.5.7 Доступ к чтению audit-логов

- Отдельная роль `audit-reader`, не пересекается с операционным админом.
- Read факт audit-лога - это сам по себе audit event ("audit-of-audit"):
  - `action: audit.log.read`
  - `actor: audit-reader`
  - `target: { type: "audit_record_range", filter: "..." }`
- Запросы из админок к audit-данным логируются с указанием пользователя и фильтра.

#### 6.5.8 Что НЕ кладём в audit-лог

- Пароли, хеши, raw токены - даже на action `password.change`. Храним метаданные: `changed_at`, `complexity_ok`, `via: web/api/sso`.
- Полные платёжные данные - храним `pan_last4`, `card_brand`.
- Содержимое токенов - храним `token_id`, `token_hash`.

Чувствительные значения при мутациях (`email`, `phone`, `name`):
- Если регулятор требует значение до/после - храним.
- Если нет - храним только факт изменения: `field_changed: email`, без значений.

#### 6.5.9 Audit и GDPR Right to Erasure

При запросе субъекта данных на удаление:
- Operational-логи удаляются/редактируются.
- Audit-логи **не удаляются** - это противоречит требованиям compliance.
- Вместо удаления - **псевдонимизация**: PII в audit заменяется стабильным идентификатором без обратимой связи с субъектом.
- Процедура документируется в DPA.

#### 6.5.10 Реализация

- Отдельный logger / channel: `Log::channel('audit')` или выделенный сервис.
- **Не** используем общий PSR-3 logger - у audit другой контракт: гарантия записи, не fire-and-forget.
- Транспорт - durable: sync в БД, или queue с at-least-once delivery. Никакого UDP.
- Сэмплинг - никогда.

> **LaraLog implementation:** в v1 audit-канал не входит - намеренно. Audit требует durable storage, что выходит за рамки операционного логирования. Рекомендуется отдельный сервис/таблица или специализированный SaaS (например, через outbox pattern + worker, пишущий в выделенную БД).

#### 6.5.11 Edge cases

**System actor.** Когда действие инициировано не человеком, а фоновой задачей/системой, `actor` всё равно обязательное поле:

```jsonc
"actor": {"type": "system", "name": "billing.daily-rollup", "trigger": "cron:0 3 * * *", "trace_id": "..."}
```

Никакого `actor: null` - аудит должен ответить, что именно это запустило.

**Cascade и bulk operations.** Одна команда меняет N записей (массовое обновление, импорт, миграция). Два варианта:

- **Per-record audit** - один event на запись. Дорого для bulk, но даёт полное отслеживание. Рекомендуется для small-mid bulk (≤1000 записей).
- **Bulk envelope** - один event с `target: {type: "bulk", count: 50000, filter: "...", sample_ids: [...]}`. Для миграций / больших операций. Sample_ids - случайная выборка для быстрой проверки.

Выбор - по регулятору. PCI-DSS обычно требует per-record для платёжных мутаций.

**Тестирование audit.** Audit-логи - часть compliance-контракта, регрессии обходятся в штрафы. Минимум:

- На каждое security-чувствительное действие в коде - тест "после вызова в audit-таблице есть запись с ожидаемыми полями".
- Снапшот-тесты на схему audit-event'а (`audit_ver`).
- Тест "audit-канал недоступен → операция падает" (не silent skip). См. §6.5.10 - audit not fire-and-forget.

**Конфликт GDPR-erasure vs регулятор.** §6.5.9 говорит "псевдонимизация вместо удаления", но в некоторых юрисдикциях/категориях (несовершеннолетние, отозванное согласие на specific категорию) регулятор может требовать именно удаления. Решение:

- Документировать в DPA, какие категории субъектов попадают под полное удаление и для каких action'ов.
- Для остальных - псевдонимизация (стабильный hash без обратимой связи).
- Любое удаление из audit само логируется в meta-audit (`action: audit.record.erase`) с правовым основанием.

**Mass export.** Скачивание оператором большого диапазона audit-записей - тоже audit-событие (§6.5.7), но дополнительно: rate-limit на размер выгрузки + двухэтапная авторизация для дампов >N записей. Защита от инсайдер-эксфильтрации.

**Hash chain vs горизонтальное масштабирование.** Линейная hash-цепочка (§6.5.5) требует одного writer'а - bottleneck на высоком throughput. Альтернативы:

- Merkle tree (партиционированная) - каждая партиция своя цепочка, корни объединяются периодически.
- External notarization - периодическое подписание hash'ей audit-batch'ей в внешнем сервисе (KMS-подпись, blockchain).

Выбор зависит от threat model - если защищаемся от внешнего атакующего, достаточно merkle. От администратора БД - нужно external.

## 7. Антипаттерны

| Антипаттерн | Почему больно | Как править |
|---|---|---|
| `log->info("user $userId logged in")` | не ищется, ломает агрегацию | `info("login success", ["user_id" => $userId, "tag" => "auth"])` |
| Логировать исключение на каждом catch-уровне | 5× шума на один сбой | catch один раз сверху, ниже - rethrow |
| `throw new SomeException()` без `previous` при пересоздании исключения | теряется цепочка причин, отладка слепая | `throw new SomeException(msg, code, $prevException)` |
| `log->error("something happened")` | бесполезно в поиске | message: существительное + глагол; context: значения |
| `log->warning(...)` на пути, который всегда срабатывает | warning fatigue | понизить до notice или убрать |
| `log->info(...)` в tight loop | заливает пайплайн | сэмплинг, агрегация per-batch или debug |
| `request_id` / `user_id` как Loki labels | взрыв индекса | как поля, не как лейблы |
| Лог полного request/response body "для отладки" | PII + взрыв размера | форма (`size`, `content_type`), не содержимое |
| Catch и swallow без лога | невидимые сбои | минимум notice/warning; лучше - пропагировать |
| `print_r($obj)` / `var_export($obj)` в message | неограниченный размер, утечка | явные поля, type coerce, truncate |
| Sync log writes на request path | один медленный Graylog = p99 наружу | UDP / async / Redis buffer (LaraLog: `RedisLogger`, capped) |
| Смешать audit и operational потоки | compliance и стоимость одновременно проигрывают | два sink'а, два retention'а |
| Warning без actionability | warning fatigue | переводить в notice |

## 8. Транспорт и пайплайн

[The Twelve-Factor App § XI - Logs](https://12factor.net/logs):

> "A twelve-factor app never concerns itself with routing or storage of its output stream. The app's environment is responsible."

На практике:

1. **App пишет в stdout/stderr** JSON-строки.
2. **Sidecar / agent** (Vector, Fluent Bit, Filebeat, syslog-ng) тейлит поток.
3. **Pipeline** (Graylog, Loki, Elastic, Datadog) делает парсинг, роутинг, retention.
4. **Backpressure** на стороне агента, не приложения.

Для low-volume / low-criticality - допустим fire-and-forget (syslog UDP, capped Redis buffer). Для audit и compliance - durable транспорт (sync, queue с persistence).

> **LaraLog recommended stack:** `['stderr']`

### 8.1 Конфиги по средам

- **Dev**: human-readable formatter, `debug`, без сэмплинга.
- **CI**: JSON, `info`, без сэмплинга, retained per-run как build artifact. **Redact env-dumps** - CI часто пишет `printenv` или dump конфигов; ban list ключей применять и тут.
- **Staging**: идентично проду, включая сэмплинг - чтобы воспроизводить volume-проблемы.
- **Production**: JSON, `info`, сэмплинг где применимо, cap per-source.

### 8.2 Sampling

Sampling - инструмент cost-control, когда объём info-логов перерастает бюджет. Применяется **только к info/debug** - notice/warning/error/critical всегда сохраняются полностью (иначе ломаются алерты §3.5).

**Head sampling.** Решение "брать или дропнуть" принимается **на источнике**, до отправки. Самое дешёвое, но контекст потерян если упало позже:

```
if (level <= INFO && hash(request_id) % 100 < sample_rate_percent) keep
else if (level >= NOTICE) keep_always
```

**Sticky-per-trace.** Решение принимается **один раз на старте trace** и распространяется во все логи этого trace. Гарантирует, что если запрос остался в выборке - вся его история на месте, не "5 логов из 12":

```
sampling_decision = hash(trace_id ?? request_id) % 100 < rate
// инжектится в MDC, переносится через очереди
```

Это де-факто стандарт для distributed tracing - то же делать для логов.

**Tail sampling.** Решение принимается **после завершения trace**, когда известен исход. Дороже (нужен буфер), но точнее: оставляем 100% trace'ов с ошибкой/высокой latency, дропаем "обычные ok-200 за 50мс". Реализуется в agent'е (OTel Collector) или в pipeline'е - не в приложении.

**Adaptive sampling.** Rate подстраивается под нагрузку: спокойно - выше rate, peak - ниже. Бюджет постоянен, выборка плотнее на анализируемых периодах.

**Метрики из логов отдельно.** Если из логов считаются метрики (error rate, p95 latency через histogram_quantile, business KPI), то sampling искажает метрики. Решения:

- Считать метрики **до** sampling (промежуточный агрегатор, который видит всё).
- Хранить sampled_rate в каждом логе - upscale при расчёте: `actual_count = sampled_count * (100 / sampled_rate)`.
- Выводить критичные метрики в Prometheus отдельно (не из логов).

**Чек-лист перед включением sampling:**

1. Алерты по error/warning сохраняются (sampling не их касается).
2. Sticky-per-trace включён (иначе теряем reconstruction).
3. Метрики, считаемые из логов, либо учитывают sampling, либо считаются до него.
4. Sampling rate отображается в логе (`sampled_rate` field или channel name).
5. Sampling не применяется к audit (никогда).

## 9. Жизненный цикл

Спецификация без ratchet'а в CI гниёт. Минимум:

1. **Линтер** (или кастомная static check): запрещает интерполяцию в `message`, `print_r`/`var_export` в логи, ban list ключей (`password`, `token`, …).
2. **Квартальный ревью**:
   - Соотношение Notice/Warning/Error по дням и неделям.
   - Топ-20 лог-строк по объёму - есть бессмысленные?
   - Топ-20 по zero-query-hits - есть неиспользуемые? Удалить.
3. **Версионирование документа.** Сервисы бампают поле `app_log_spec` (например, `"v1"`) - postmortems знают, какие правила действовали.
4. **Онбординг.** Новые инженеры читают этот документ на день 1. Новые лог-строки в PR проверяются против §7 коллегой.

### 9.1 Метрики, за которыми следим

- **MTTR** - главная. Если MTTR не падает по мере роста логов - логи неправильные.
- **Notice/Warning/Error rates** - должны быть стабильными. Скачок одного из них - сигнал.
- **Стоимость лог-пайплайна** ÷ **трафик** - плоская или падает. Растёт = логируем то, что не читаем.
- **Signal/noise** в Warning: % warning'ов с actionable действием. Меньше 50% - пора прибираться (демоут до notice или удаление).
- **Drop rate** агента (Vector/Fluent Bit metrics) - должен быть около нуля. Растёт = backpressure, апплоадить или дропать осознанно.
- **Schema violation rate** - % логов без обязательных top-level полей. Должен быть 0 после миграции, всплеск = регрессия в новом сервисе.

### 9.2 Тестирование логов

Линтер (§9 п.1) - reactive. Тесты - proactive. Минимум:

**Unit-тесты:** проверяем, что код пишет лог нужного уровня с нужными полями.

```php
// PHP / Monolog
use Monolog\Handler\TestHandler;

$handler = new TestHandler();
$logger = new Logger('test', [$handler]);
$service = new BillingService($logger);

$service->chargeFailed($order);

$this->assertTrue($handler->hasErrorRecords());
$record = $handler->getRecords()[0];
$this->assertSame('charge failed', $record['message']);
$this->assertSame('billing', $record['context']['tag']);
$this->assertSame($order->id, $record['context']['order_id']);
```

**PII regression тесты:** для каждой точки, где может проскочить чувствительное значение, явный тест:

```php
public function test_login_failure_log_does_not_contain_password(): void
{
    $this->postJson('/login', ['email' => 'a@b.c', 'password' => 'secret123']);
    $logs = $this->handler->getRecords();
    foreach ($logs as $log) {
        $serialized = json_encode($log);
        $this->assertStringNotContainsString('secret123', $serialized);
    }
}
```

Стоит держать generic test, который для **списка известных секретов** проверяет, что они никогда не попадают в логи в smoke-сценариях. Дёшево, ловит регрессии в редакторах.

**Integration-тест пайплайна** (не на каждый PR, но на инфра-changes):

- Поднять docker-compose с приложением + агентом + Graylog/Loki.
- Сгенерировать N логов разного уровня.
- Проверить, что в backend пришли N записей с правильной схемой.
- Проверить, что один из них с PII был **отредачен на этапе агента** (если редактирование там).

**Schema snapshot.** Снапшот реального лог-объекта с замазанными datetime'ами. PR, меняющий схему, "ломает" снапшот и форсит явное ревью изменения контракта.

### 9.3 Каталог полей и schema lifecycle

Документ-без-каталога - это рекомендация без enforcement. Каталог делает поля **discoverable** и **mutable безопасно**.

**Каталог полей.** Один файл (`docs/log-fields.yml` или подобный) - source of truth: какое поле есть, в каком слое (top-level/extra/context), тип, описание, kim используется, deprecation status.

```yaml
- name: order_id
  layer: context
  type: int
  description: Идентификатор заказа в billing-домене
  owner: team-billing
  added_in: v1
- name: payment_method_old
  layer: context
  type: string
  status: deprecated
  deprecated_in: v3
  remove_in: v5
  replaces_with: payment_method  # имя нового поля
```

Каталог - часть PR-ревью: новое поле = новая запись в каталоге.

**Whitelist для `tag`.** `tag` - индексируемый лейбл (см. §5.5), бесконтрольное добавление = cardinality explosion. Whitelist в каталоге:

```yaml
tags_allowed:
  - auth
  - billing
  - sql
  - queue
  - upstream
  - entrypoint
  - audit
  - cron
```

Линтер проверяет, что `context.tag` в каждом логе - из whitelist. Новый tag = PR-ревью + явное добавление.

**Lifecycle поля.**

| Стадия         | Что делаем                                                                                  |
|----------------|---------------------------------------------------------------------------------------------|
| `proposed`     | поле в каталоге со статусом `proposed`, ещё не пишется кодом                                |
| `active`       | пишется и используется в дашбордах/алертах                                                  |
| `deprecated`   | пишется параллельно с заменой (dual-write window ≥ 2 релиза)                                |
| `read-only`    | новые записи без него, старые остаются для исторического поиска                             |
| `removed`     | поле удалено из каталога; runbook'и и дашборды должны быть мигрированы заранее              |

**Переименование = dual-write.** Прямое переименование ломает поиск, dashboards, alerts. Корректный workflow:

1. Каталог: добавить новое поле `active`, старое перевести в `deprecated`.
2. Код: писать оба поля.
3. Migrate dashboards/alerts на новое поле.
4. Через N релизов: каталог - старое `read-only`, код - перестать писать.
5. Через ещё N релизов: `removed`.

**Версионирование контракта.** Top-level поле `app_log_spec: "v3"` бампается при breaking change'ах схемы. Postmortems по старым инцидентам знают, какой контракт действовал. На каждый бамп - запись в changelog документа.

#### 9.3.1 OpenTelemetry semantic conventions

Структурно совместимо: `extra` соответствует OTel `Resource` (per-service stable), `context` - OTel `Attributes` (per-event), `message` - `Body`, `datetime` - `Timestamp`, `level`/`level_name` - `SeverityNumber`/`SeverityText`.

Имена полей расходятся с [OTel Semantic Conventions](https://opentelemetry.io/docs/specs/semconv/) осознанно - в пользу краткости и удобства запросов в Loki/Graylog (точки в OTel-именах ломают парсеры). При интеграции с OTel-стеком (Tempo, Honeycomb, Datadog) - конвертация через OTel Collector `transform` processor или Vector `remap`.

| LaraLog                | OTel sem conv                       | Заметка                              |
|------------------------|-------------------------------------|--------------------------------------|
| `datetime`             | `Timestamp`                         |                                      |
| `level`                | `SeverityNumber`                    |                                      |
| `level_name`           | `SeverityText`                      |                                      |
| `message`              | `Body`                              |                                      |
| `channel`              | -                                   | нет прямого аналога                  |
| `app_name`             | `service.name`                      |                                      |

| `app_env`              | `deployment.environment`            |                                      |
| `tier`                 | `service.namespace`                 | приблизительно                       |
| `host_name`            | `host.name`                         |                                      |
| `host_ip`              | `host.ip`                           |                                      |
| `container_name`       | `container.name`                    |                                      |
| `trace_id`, `span_id`  | `trace_id`, `span_id`               | совместимы (snake_case)              |

Новые поля - при добавлении в каталог §9.3 сразу проверяй, нет ли OTel-эквивалента; если есть - либо принять OTel-имя, либо явно зафиксировать divergence в каталоге.

## 10. Быстрый старт (LaraLog)

При использовании LaraLog уже есть:

- Авто-инжект `request_id`, `app_ver`, `host`, memory-метрик, release-инфы (`ExtraProcessor`).
- Cap'ы на размер message и целой строки со smart-truncation (`LogManager::message_limit`, `CustomFormatter`).
- Exception unwrap с previous-цепочкой (`appendContext`).
- Type coercion в context (`contextTypeCorrector`).
- File:line точки вызова, без framework-кадров (`TraitFileTrace`).
- Авто stack trace на `Warning+`, глубина по уровню (5/10/20).
- Slow SQL канал с безопасной редакцией биндингов (`SqlLogServiceProvider`).
- Queue heartbeat в лог (legacy, мигрируем на Prometheus).
- Каналы: stderr, syslog-UDP → Graylog, JSON-файл, Redis (capped), Telegram.

Рекомендуемая композиция:

```php
'stack' => [
    'driver' => 'stack',
    'channels' => ['stderr'],
],
```

Что всё равно на тебе:

- Правильный **уровень** в точке вызова (§3).
- **Стабильные имена полей** + бизнес-ID в context (§4.4).
- Соблюдение **списка редактирования** (§2).
- **Cardinality** под контролем в label-схеме (§5.5).
- **Audit-канал отдельно** (§6.5).

### 10.1 Миграция с Monolog default

Если у тебя сейчас Monolog без LaraLog (или Laravel default `LineFormatter`):

1. **Включить JSON-формат на проде** - заменить `LineFormatter` на `JsonFormatter` для `stderr`/`syslog` каналов. Это минимально-инвазивный шаг.
2. **Добавить `ExtraProcessor`** (LaraLog): подтянет `app_*`, `host`, `release_*` без правок в коде приложения.
3. **Подключить `LogManager` через `Providers/AppServiceProvider`** - тогда автоматически добавляется trace, exception_prev-цепочка, type coercion.
4. **Перевести точки логирования на структуру** (`['key' => $val]` вместо `"... $val ..."`) - проверять линтером в CI, делать поэтапно по доменам.
5. **Добавить `tag`** в whitelist каталога (§9.3) и инжектить в каждый лог.
6. **Включить slow SQL канал** (`SqlLogServiceProvider`) последним - до этого pipeline уже устаканен.

Параллельно с миграцией добавить в каталог поля старый и новый формат, dual-write пока dashboards переезжают.
